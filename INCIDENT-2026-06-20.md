# Security Incident — 2026-06-20

## Summary

The original VPS (`46.225.186.103`) was compromised with an XMRig crypto miner.
A new VPS was provisioned (`89.167.93.25`) but the attacker re-entered the new server
within 2 hours using the GitHub Actions SSH private key stolen from the old server.

## Root Cause

The GitHub Actions SSH private key (`github-actions-liveportfolio`) was stored in
`authorized_keys` on the old compromised server. The attacker extracted it and used it
to authenticate to the new server before it could be rotated.

**Key lesson: Never carry SSH keys from a compromised server to a new one.**
Generate new keys first, add them to the new server, then do the migration.

## Mistakes Made During Response

1. Did not immediately rotate the GitHub Actions SSH key before setting up the new server
2. Ran `npm audit fix --force` on the VPS directly — downgraded Next.js to 9.3.3, breaking the app
3. Did not check `root` authorized_keys until late in the session
4. Did not lock `authorized_keys` with `chattr +i` immediately on server setup

---

## Prevention for Next Server Setup

1. Generate ALL new SSH keys before provisioning a new server after a compromise
2. Lock `authorized_keys` with `chattr +i` immediately after adding keys
3. Set `PermitRootLogin no` on first login
4. Mount `/tmp` and `/var/tmp` as `noexec` to prevent miner execution
5. Never run `npm audit fix --force` on a production server
6. Rotate all secrets immediately when a server is confirmed compromised
